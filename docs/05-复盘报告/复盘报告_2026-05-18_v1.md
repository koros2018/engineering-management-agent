### 3.1 继承式开发策略

**策略**: EMA 不重复造轮子，直接复用 blueprint-ai 成熟模块

| Sub-Agent | 复用模块 | 效果 |

| CostBenefitAgent | `budget.py` / `documents.py` | 工程量提取 + 预算生成 |

**效果**: EMA Phase 1-3 只用了 16 个 commit，但覆盖了 6 个 Sub-Agent 的核心能力

### 3.2 懒加载避免启动阻塞

**问题**: ChromaDB 初始化时下载 79MB ONNX 模型，阻塞 API 启动数分钟

**方案**:
```python
@property
def _chroma_store(self):
    if self._chroma is None:
        from memory import get_chroma_store
        self._chroma = get_chroma_store()
    return self._chroma
```

**效果**: API 启动从"等待 ONNX 下载"变为"秒级就绪"，ONNX 在后台慢慢下载

### 3.3 后台线程存储对话历史

**问题**: ChromaDB 写入阻塞 API 响应，导致请求超时

**方案**:
```python
import threading
def _store():
    chroma.add_conversation(...)
threading.Thread(target=_store, daemon=True).start()
```

**效果**: API 响应不受 ChromaDB 影响，对话历史在后台异步存储

### 3.4 UI 增强的分层策略

**Phase 2 UI 改进**:
1. **结构层**: 侧边栏 Agent 列表 → 点击切换
2. **展示层**: 结果卡片分类展示（图纸分析 / 审图报告 / 文档生成）

## 四、教训与反思

### 4.1 ChromaDB ONNX 下载阻塞问题

**现象**: API 启动时 ChromaDB 尝试初始化 embedding 模型，下载 79MB ONNX 文件，速度极慢（~10kiB/s）

**影响**:
- API 启动后数分钟内请求阻塞
- 旧版本 API 进程假死

**根因**: `get_chroma_store()` 在 Main-Agent `__init__` 中被调用，触发了 ChromaDB 客户端初始化

**修复**:
- 改为 `@property` 懒加载
- ChromaDB 在首次 `_chroma_store` 访问时才初始化

**教训**: **任何可能阻塞初始化的大资源（模型下载/数据库连接/网络请求）必须懒加载**

---

### 4.2 IntentClassifier 默认 task_type 误判

**现象**: 用户说"你好"（无文件），系统识别为 `tech_rd:full_analysis`，要求上传图纸

**根因**:
- IntentClassifier 对无文件消息默认返回 `full_analysis`
- TechRdAgent 的 `full_analysis` 必须有 `file_path`
- TaskPlanner 的 fallback 也是 `full_analysis`

**修复**:
- tech_rd 默认 task_type 改为 `chat`
- greeting 关键词（你好/hi/hello/介绍）直接返回 chat
- TechRdAgent 新增 `chat` task_type 处理

**教训**: **意图分类要考虑"无文件"的闲聊场景，不能假设每个请求都有文件**

---

### 4.3 TaskPlanner fallback 逻辑缺陷

**现象**: `"你好"` → intent = `tech_rd:chat` → 但 TaskPlanner 的 SINGLE_STEP_TASKS 中没有 `tech_rd:chat` → fallback 到 `tech_rd:full_analysis`

**根因**: TaskPlanner 的 fallback 逻辑是硬编码的默认路径，没有检查模板是否存在

**修复**: 在 IntentClassifier 层面保证默认 task_type 有对应的执行路径

**教训**: **Planner 的 fallback 不能依赖"默认某个 task_type"，必须有明确的兜底策略**

---

### 4.4 多阶段修复的重复劳动

**现象**: ChromaDB 问题在 Phase 2 出现，修复后在 Phase 3 再次出现相同问题（ONNX 阻塞 API）

**根因**: 修复时只处理了 Main-Agent 的 `_chroma_store`，但 API Server 启动时其他位置也调用了 get_chroma_store()

**教训**: **修复问题要看完整调用链，不能只修表面调用点**

---

### 4.5 UI 服务端口管理问题

**现象**: Windows 用户启动 EMA 时，UI 服务（5189）端口被旧进程占用，导致重启后 UI 无法访问

**根因**: Windows BAT 文件没有清理旧进程的能力

**改进**: 启动脚本添加端口检测 + 旧进程清理

**教训**: **Windows 批处理要做进程管理，不能只管启动不管冲突**
